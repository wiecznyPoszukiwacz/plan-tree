*** QUESTION Konwencja zapisu opcji — strict format czy luźny parser?
:PROPERTIES:
:ID: n65
:END:
Dziś moje QUESTION mają opcje "(A) ..." / "(B) ..." (patrz n52, n59, n61) — konwencja organicznie się ustaliła, ale nie wymuszona.
Opcje formatu:
(A) Strict: linia musi zaczynać się "(X) " gdzie X = pojedyncza litera A-Z. Wszystko inne ignorowane przez parser.
(B) Luźny: parser akceptuje "(A)", "A)", "A.", "- A:", "* A —", "[A] " — kilka popularnych form.
(C) Strict + dedykowana sekcja: notes muszą mieć blok "Opcje:" a pod nim linie "(A) ...". Reszta notes (Rekomendacja, kontekst) poza tym blokiem.
Rekomendacja: (C). Powody:
- Eliminuje fałszywe trafienia (np. "(A)" wewnątrz cytatu nie zostanie zinterpretowane jako opcja).
- Wymusza ode mnie dyscyplinę — i tak piszę "Opcje:" naturalnie, samo skodyfikowanie jest tanie.
- Łatwiej dorzucać meta-informacje per opcja (np. "(A) ... [tag:risky]" — parser może to wykorzystać).
Powiązane: [[n51]] (format wpisu inbox) — tam celowo luźno; tu strict, bo to ja piszę a nie user.

*** QUESTION Co po wyborze — co dzieje się z QUESTION i jak agent dostaje sygnał
:PROPERTIES:
:ID: n67
:END:
Po naciśnięciu klawisza A/B/C:
Krok 1 — natychmiast w TUI:
- Dodaj wpis pod [[n49]]: title np. "Decyzja n_NN: opcja B" (gdzie n_NN to ID QUESTIONa), notes = cytat treści opcji + ewentualnie pełny kontekst pytania.
- Opcjonalnie tag :choice:.
Krok 2 — co z oryginalnym QUESTION:
(A) Zostaje QUESTION, niczego nie zmieniamy — agent dopiero zamknie go na DONE po zaaplikowaniu konsekwencji.
(B) Od razu setTodo na nowy stan np. ANSWERED lub DONE z notatką "user wybrał B" — agent widzi że jest po sprawie, ma tylko zaaplikować.
(C) Hybrid: setTodo zostaje QUESTION ale dodaje property :USER-CHOICE: B — sygnał dla agenta, ale wciąż widoczne że czeka na obsługę.
Rekomendacja: (C). Powody: (i) nie traci się informacji że to było pytanie; (ii) agent widzi gotową decyzję bez czytania całego inbox; (iii) property łatwo czyścić po zaaplikowaniu (wraz z setTodo DONE).
Krok 3 — sygnał do agenta:
Spójne z [[n52]] (kiedy agent czyta inbox): TUI nic nie pushuje do agenta proaktywnie; przy następnej sesji/wezwaniu agent widzi inbox count + zaaplikuje. Jeśli user chce natychmiast — w czacie "obsłuż n49".
Edge case: user wybrał, ale potem chce zmienić. Wpis w inbox nie został jeszcze obsłużony — może go usunąć (delete na wpisie inbox) i wcisnąć inny klawisz na QUESTION (TUI nadpisuje property :USER-CHOICE: i tworzy nowy wpis inbox). Jeśli już obsłużony — zwykły flow zmiany planu.

*** QUESTION Relacja z klawiszem R z n32 — kiedy R, kiedy A/B/C
:PROPERTIES:
:ID: n69
:END:
[[n32]] mówi: R = akceptacja "Rekomendacji" z notes (rename + DONE w jednym ruchu). To była wizja dla QUESTION typu binary ("zrobić X? / Rekomendacja: tak").
Ten feature dodaje multi-choice (A/B/C/...). Pytanie: czy R wciąż istnieje osobno, czy zostaje wchłonięte?
Opcje:
(A) R = "zaakceptuj rekomendację" działa dla obu przypadków:
    - QUESTION z jedną sugestią "Rekomendacja: tak/nie/zrób X" → R wykonuje.
    - QUESTION z opcjami A/B/C i notatką "Rekomendacja: opcja B" → R = równoważne wciśnięciu B.
(B) R i A/B/C koegzystują rozłącznie — R tylko dla binary, A/B/C tylko dla multi-choice. Parser decyduje który tryb aktywny.
Rekomendacja: (A). R = uniwersalny "ufam ci, zrób co rekomendujesz". Mentalnie spójne: jedna konwencja "wpisuję 'Rekomendacja: ...' w notes, user ma R do akceptacji". Multi-choice tylko rozszerza co może być rekomendowane (od "tak/nie" do "opcja B"). Implementacja: parser szuka "Rekomendacja: opcja X" lub "Rekomendacja: <tekst>" — w pierwszym wypadku symuluje wybór X, w drugim — to co dziś n32.
*** QUESTION Parser może źle zinterpretować notes nie będące opcjami :risk:
:PROPERTIES:
:ID: n70
:END:
Ryzyko false positive: notes zawiera "(A) coś tam" jako cytat / dygresję / przykład, parser pokazuje opcje gdzie ich nie ma, user wciska A → tworzy się wpis w inbox dla "wyboru" którego nigdy nie było.
Mitygacje (kombinujemy):
1. Strict format z dedykowaną sekcją "Opcje:" (rekomendacja z n65) — eliminuje 90% false positives.
2. Wymóg minimum 2 opcji żeby parser uznał że to multi-choice (samotne "(A)" w środku tekstu nie aktywuje trybu).
3. Wszystkie opcje muszą być w jednym ciągłym bloku linii (przerwa = koniec listy opcji).
4. Wizualne potwierdzenie przed zapisem do inbox: po wciśnięciu A pokaż na 1s "Wybrano A. Enter=potwierdź, Esc=anuluj" — daje sekundę na cofnięcie.
Rekomendacja: 1+2+3 obowiązkowo, 4 opcjonalnie (może być irytujące jeśli user już ufa). Domyślnie 4 włączone, klawisz w settings żeby wyłączyć dla power-userów.
** PROPOSAL Agent-inbox + blokada add na root/N-poziomach — twardy hamulec na scope creep ze strony agenta
:PROPERTIES:
:ID: n71
:END:
Symetria do [[n49]] (user-inbox), ale w drugą stronę. Cel: zmusić mnie do triage zanim wrzucę pomysł do struktury planu.
Problem dziś: gdy mam pomysł, mam dwie ścieżki — (1) wrzucić jako top-level lub child gdzieś "tam pasuje", (2) wspomnieć w czacie. (1) to scope creep. (2) ginie. Brakuje trzeciej: "miejsce parking dla pomysłów które nie zasłużyły jeszcze na strukturalne miejsce".
Mechanizm — trzy filary:
1. **Agent-inbox** (osobny węzeł, np. n_AI). Wrzucam tam pomysły które:
   - Nie pasują do żadnego istniejącego rodzica.
   - Są poniżej importance threshold zadanego przez usera.
   - Nie mam pewności gdzie powinny trafić.
2. **Importance threshold** trzymany przez MCP server. User ustawia (np. setThreshold(7) skala 1-10 lub "high"/"medium"/"low"). MCP echo'uje threshold w każdej odpowiedzi tool/resource (jak ambient `selection`) — nie mogę "zapomnieć". Pomysł poniżej thresholdu → musi iść do agent-inbox.
3. **Blokada add na poziomie roota i N poziomów w dół.** MCP odrzuca `add(parentId=root, ...)` (i głębiej do depth N) z błędem "Use agent-inbox or escalate to user — depth-locked". User ma osobny tool/klawisz żeby tymczasowo odblokować ("user-grant: dodaj jako top-level").
Wynik: nawet jeśli moja heurystyka importance zawiedzie, twarda blokada strukturalna nie pozwoli mi rozsadzić planu. Pomysł trafia do agent-inbox, user widzi tam wszystko co pomyślałem, sam decyduje co promować do struktury.

- (i) Liczba 1-10 (precyzyjnie ale trudniej skalibrować "co znaczy 7").
- (ii) Słowa: low / medium / high / critical (mniej precyzyjne ale natychmiast czytelne).
- (iii) Hybrid: 4 poziomy z opisami ("low = ciekawostka", "high = blokuje decyzję").
Rekomendacja: (ii) z 4 poziomami. Konkretne i pamiętalne. User ustawia `setThreshold("high")` raz na sesję, MCP echo'uje w każdym responsie.
B) Jak agent ocenia importance własnego pomysłu:
Heurystyka samoocenowa:
- critical: blokuje aktualne zadanie, nieuwzględnione = fatal.
- high: znacząco zmienia decyzję/architekturę, ale można pracować bez tego.
- medium: ulepsza/uściśla istniejący kierunek.
- low: ciekawostka, dygresja, alternatywa do rozważenia "kiedyś".
Default agent = low (konserwatywnie). Pomysł > threshold → agent może dodać do struktury jeśli pasuje (lub user-grant). Pomysł ≤ threshold → MUSI iść do agent-inbox.
Edge: gdy nie potrafię ocenić — domyślnie low (= agent-inbox). Bezpieczniej dla scope.

*** QUESTION Blokada add — głębokość N, zakres, bypass
:PROPERTIES:
:ID: n75
:END:
Trzy decyzje:
A) Co dokładnie blokujemy:
- (i) Tylko add z parentId="root" (samo top-level).
- (ii) Top-level + N poziomów (user-konfigurowalne, np. 2 = root i jego dzieci).
- (iii) Cała ścieżka od roota do węzła z `:OPEN-FOR-AGENT: t` property — agent może dodawać tylko poniżej wyraźnie otwartych miejsc.
Rekomendacja: (ii) z konfigurowalnym N (default 1 = tylko root, czyli minimum). User może podnieść do 2-3 dla planów gdzie chce dyscypliny też na pod-tematach. Property :ADD-LOCK-DEPTH: N na rooсie.
B) Co znaczy blokada:
- Hard: MCP zwraca error "depth-locked, use agent-inbox or request user-grant". Agent nic nie wymusza.
- Soft: ostrzeżenie w response, ale add przechodzi. Agent ma decydować.
Rekomendacja: Hard. Cały sens to strukturalne uniemożliwienie, nie apel do dobrej woli.
C) Bypass — kiedy agent może legalnie:
- User-grant: tool `grantAddPermission(parentId, ttlSeconds=60)` — user wykonuje, MCP zapamiętuje na N sekund że agent ma jednorazowy(/krótkoterminowy) grant na ten parent.
- Explicit user prompt w czacie "dodaj jako top-level X" — agent woła grant + add. User widzi w response confirmation.
- Agent NIGDY nie grantuje sam sobie.
Rezultat: ścieżka pomysł→struktura zawsze przechodzi przez user (grant lub triage z agent-inbox).
**** PROPOSAL Trzeci tryb bypass: synchroniczny modal potwierdzenia w TUI
:PROPERTIES:
:ID: n79
:END:
Dodatkowa opcja do hard block + grant (nie zamiast). Gdy agent woła `add` na zablokowanym poziomie, plan-tree TUI wyświetla modal:


- Po timeout MCP zwraca default akcję — opcje:
  (i) Hard block ("brak reakcji = nie wolno", agent dostaje error i idzie do agent-inbox sam).

- Property `:CONFIRM-ADD: t` na konkretnym rodzicu — agent add pod tym rodzicem zawsze pyta.
- Globalnie: `:ADD-CONFIRM-DEPTH: N` na rooсie — analogicznie do `:ADD-LOCK-DEPTH:`.
Kombinacja z hard lock: jeśli oba ustawione, lock wygrywa (modal nie pokazany, error od razu). Jeśli tylko confirm — modal się pokazuje. Jeśli tylko lock — bez modalu.
Edge case: agent dodaje batch (np. rozpakowuje QUESTION na 5 children). Modal per child = 5 pytań = nuke productivity. Mitigation: gdy agent woła `add` w krótkim oknie (np. <2s od poprzedniego), modal oferuje dodatkowo `[S]kip rest (auto-accept this batch)` — user raz akceptuje całą falę.
Powiązane: [[n67]] (wpis w inbox z wyborem A/B/C) używa identycznego mechanizmu modal+klawisz — warto zunifikować komponent UI.

:PROPERTIES:

** QUESTION Zakres egzekwowania: tylko anchor sam czy też jego dzieci?
:PROPERTIES:
:ID: n46
:END:
Opcje:
(A) Tylko sam anchor jest chroniony — dzieci anchora można dowolnie zmieniać. Prostsze, ale anchor "naprawa autoryzacji" mógłby mieć zmieniane dzieci przez agenta — sens?
(B) Anchor i całe poddrzewo chronione — żelazna kotwica. Ale zmniejsza elastyczność (anchor wysoko → ogromny obszar zamrożony).
(C) Anchor pływający — tylko bezpośrednie pole title/notes/todo węzła, ale agent może DODAWAĆ dzieci, MOVE-ować je. Tak żeby anchor był "punktem orientacyjnym" a nie zamrożeniem poddrzewa.
(D) Dwa rodzaje: anchor-self (tylko węzeł) i anchor-subtree (poddrzewo) — najelastyczniej, ale więcej kompleksu.
Rekomendacja: (C) jako default, plus opcjonalny tag :anchor-subtree: dla rzadkich przypadków pełnego zamrożenia.
** QUESTION Czy agent może unanchor na explicit user request, czy tylko user?

* Inbox — skrótowe notatki ode mnie, agent rozwija i przenosi
:PROPERTIES:
:ID: n49
:END:
Jedno miejsce, gdzie wrzucam jednolinijkowe myśli/pomysły/feedbacki w trakcie pracy — bez zastanawiania się gdzie pasują w strukturze. Agent (Claude) jest odpowiedzialny za triage: czyta każde dziecko n49, rozumie intencję, uzupełnia szczegóły, przenosi pod właściwego rodzica w drzewie.
Założenie: tempo dorzucania > tempo strukturyzowania. Inbox jest zaworem bezpieczeństwa przeciw "zapomniałem, bo nie miałem gdzie tego wrzucić".
Po przeniesieniu wpis znika z inbox sam (move = zmiana rodzica). Inbox jest pusty = wszystko obsłużone.
Protokół i decyzje — patrz dzieci.
** PROPOSAL Format wpisu — title = jedna myśl, notes opcjonalne
:PROPERTIES:
:ID: n51
:END:
Rekomendacja: title to dosłownie to, co user wpisze (krótka myśl), bez wymuszania struktury. Notes opcjonalne — jeśli user dorzuci kontekst, super; jeśli nie, agent dopytuje albo zgaduje.
Bez TODO state na nowych wpisach (default TODO jest OK) — agent nadaje właściwy stan przy triażu (QUESTION jeśli to pytanie do agenta, PROPOSAL jeśli to pomysł na funkcję, RISK jeśli ostrzeżenie, itd.).
Alternatywa odrzucona: wymuszanie prefiksów typu "Q: ...", "P: ..." — zwiększa friction wrzucania, niwelując sens inbox.
** QUESTION Kiedy agent czyta inbox?
:PROPERTIES:
:ID: n52
:END:
Opcje:
(A) Tylko na explicit "przejrzyj inbox" / "obsłuż n49".
(B) Proaktywnie — na początku każdej dłuższej sesji agent sprawdza czy są dzieci n49 i pyta czy obsłużyć.

** PROPOSAL Protokół przetwarzania pojedynczego wpisu
:PROPERTIES:
:ID: n53
:END:
Rekomendowany cykl per wpis:
1. Read title + notes.
2. Klasyfikuj: task/proposal/question/risk/feedback/decyzja-do-podjęcia/duplikat-czegoś-co-już-jest.
3. Znajdź właściwego rodzica w drzewie (find po słowach kluczowych, sprawdź czy istnieje podobny węzeł).
4. Jeśli duplikat / rozwinięcie istniejącego: absorb pod istniejący węzeł (zachowaj oryginał usera w notes).
5. W innym wypadku: rename (rozwiń skrót do pełnego zdania), setTodo (właściwy stan), setNotes (dodaj kontekst/rekomendację/uzasadnienie), opcjonalnie addTag, na końcu move pod właściwego rodzica.
6. Po move wpis znika z inbox — done.
Zasada zachowania: nigdy nie usuwaj oryginalnego brzmienia usera. Jeśli rename zmienia title — wrzuć oryginał do notes jako "Oryginał z inbox: ..." żeby user mógł zweryfikować że interpretacja jest OK.
Powiązane: [[n31]] (stan ASSUMED) — przeniesienie pod nowy temat powinno być ASSUMED dopóki user nie potwierdzi.
** QUESTION Co gdy wpis to pytanie/feedback dla agenta, nie task?
:PROPERTIES:
:ID: n54
:END:
Przykład wpisu: "ciekawi mnie czy klawisz R z n32 miałby konflikt z 'r' jako rename?".
To nie task — to pytanie do mnie wymagające odpowiedzi tekstowej, nie ruchu w drzewie.
Rekomendacja: agent odpowiada w czacie + wrzuca odpowiedź jako notes do tego wpisu, zmienia title na bardziej deklaratywny ("Konflikt R/r — analiza"), setTodo QUESTION (jeśli to wciąż wymaga decyzji usera) lub DECISION (jeśli odpowiedź zamyka temat), i przenosi pod właściwego rodzica (np. n32 jeśli o klawisz R). Nie zostawiaj w inbox — inbox jest staging area, nie archiwum.
** QUESTION Agent może źle zinterpretować skrótową notatkę i ulokować ją w złym miejscu :risk:
:PROPERTIES:
:ID: n55
:END:
Im krótszy wpis, tym większa szansa na błędną interpretację. Skutek: wpis ląduje pod złym rodzicem, user nie widzi go bo nie scrolluje tam, kontekst się gubi.
Mitygacje (kombinujemy):
1. Zawsze ASSUMED ([[n31]]) po przeniesieniu — user widzi że agent się domyślał, jeden klik cofa.
2. W notes obowiązkowo "Oryginał z inbox: ..." + "Domysł interpretacji: ..." — user może zweryfikować rozumowanie.
3. Przy niejasnym wpisie agent NIE przenosi — zostawia w inbox, zmienia na QUESTION z rekomendacją interpretacji w notes, czeka na 'R'.

* Rock-solid (RS) — konsensus dominujący, edytowalny tylko za zgodą usera
:PROPERTIES:
:ID: n50
:END:
RS i ANCHOR ([[n42]]) to ten sam mechanizm "punktów chronionych": dominują nad non-protected, triggerują audyt, konflikty między protected (ANCHOR↔ANCHOR, RS↔RS, ANCHOR↔RS) rozstrzyga user.
Jedyna różnica: kto może edytować treść punktu.
- ANCHOR: tylko user. Agent nigdy nie proponuje zmiany.
- RS: user (zawsze) + agent (tylko za explicit zgodą — patrz [[n62]]).
Wszystkie zasady niżej (dominacja nad non-protected, audyt, eskalacja konfliktów) stosują się jednakowo do ANCHOR i RS. W tekście dzieci "RS" = "RS lub ANCHOR" chyba że wyraźnie zaznaczono inaczej.
Cel RS jako rozszerzenia ANCHOR: dać userowi punkty fundamentalne, które są otwarte na rewizję jeśli agent znajdzie dobry argument, ale bez ryzyka cichej zmiany. ANCHOR pozostaje opcją "nawet nie próbuj".
Decyzje fundamentalne — patrz dzieci DONE. Otwarte pytania — QUESTION/PROPOSAL.
** DONE Decyzja: konflikt protected (RS/ANCHOR) ↔ non-protected — protected wygrywa :decision:
:PROPERTIES:
:ID: n56
:END:
Gdy non-protected punkt (zwykły) jest niespójny z jakimś RS lub ANCHOR, dostosowywany jest non-protected — protected stoi. Agent nie pyta usera o pozwolenie na zmianę non-protected w tym kontekście; informuje co zrobił.
Powód: protected jest definicją "tego nie negocjujemy w tym poddrzewie". Każda inna interpretacja niwelowałaby semantykę.
Edge case: jeśli zmiana non-protected jest na tyle drastyczna że zmienia jego sens (trzeba go usunąć / DROPPED / przeformułować od zera) — agent przenosi non-protected w stan QUESTION z notatką "konflikt z [[nNN]] (protected), propozycja: zdroppować lub przeformułować jako X" i eskaluje do usera. Granica: zmiana mechaniczna = agent, zmiana semantyczna = user.
** DONE Decyzja: pojawienie się nowego protected (RS/ANCHOR) triggeruje audyt non-protected :decision:
:PROPERTIES:
:ID: n57
:END:
Trigger: każde nadanie flagi RS lub ANCHOR węzłowi (świeży protected lub promocja istniejącego non-protected). Jeśli istniejący RS/ANCHOR zostaje edytowany (RS przez agenta z consent, ANCHOR przez usera) — też trigger audytu, bo zmiana ograniczeń = możliwe nowe konflikty.
Akcja agenta: przejdź po wszystkich non-protected węzłach w zakresie audytu ([[n61]]), oceń spójność z nowym/zmienionym protected, dla każdego konfliktu wykonaj politykę z [[n56]]. Wynik audytu: raport w czacie ("audyt nNN: 2 konflikty rozwiązane automatycznie [nXX, nYY], 1 eskalowany [nZZ]") + tagi/notes na zmienionych węzłach żeby user mógł zweryfikować.
** DONE Decyzja: konflikt między protected (RS↔RS, ANCHOR↔ANCHOR, RS↔ANCHOR) rozstrzyga user :decision:
:PROPERTIES:
:ID: n58
:END:
Gdy agent zauważy że dwa protected punkty (dowolna kombinacja RS i ANCHOR) są wzajemnie sprzeczne, NIE wybiera za usera. Eskaluje:
- Dodaje QUESTION (np. "Konflikt protected [[nXX]] ↔ [[nYY]]") z notatką: oba teksty zacytowane, opis sprzeczności, ewentualnie propozycja rozstrzygnięcia (jeśli to RS↔RS i agent ma rekomendację — w przypadku ANCHOR agent nie sugeruje rozstrzygnięcia, czeka biernie).
- Sygnalizuje w czacie żeby user nie przeoczył.
- Wstrzymuje wszystkie akcje zależne od któregokolwiek z konfliktujących protected aż user rozstrzygnie.
Powód: protected jest fundamentem. Cicha decyzja agenta o tym który "wygrywa" niszczy zaufanie do mechanizmu.

Konsekwencje:

** QUESTION Zakres audytu — całe drzewo czy poddrzewo?
:PROPERTIES:
:ID: n61
:END:
Gdy RS pojawia się na n42, agent audytuje:
(A) Całe drzewo (kosztowne dla planów 100+ węzłów, ale gwarantuje spójność globalną).
(B) Tylko poddrzewo n42 i jego rodzeństwo (tańsze, ale RS może mieć implikacje poza swoim poddrzewem).
(C) Heurystycznie: agent na podstawie treści RS sam decyduje co jest "w pobliżu semantycznie" (np. find po słowach kluczowych z RS).
Rekomendacja: (C) z fallbackiem na (A) gdy user explicit ("audytuj wszystko vs RS n42"). W praktyce: agent po promocji RS pisze w czacie "audytuję X węzłów semantycznie powiązanych", user może rozszerzyć.
Argument przeciw (A) zawsze: hałas — większość węzłów nie jest powiązana z konkretnym RS, audyt zwróci pustkę dla 90%, ale pochłonie kontekst i czas.
** PROPOSAL Protokół edycji RS za zgodą usera
:PROPERTIES:
:ID: n62
:END:
Gdy agent chce zmienić RS (bo np. pojawia się argument który ją podważa lub doprecyzowuje), NIE robi tego cicho. Cykl:
1. Agent dodaje pod RS dziecko QUESTION: "Propozycja zmiany RS: <opis> — powód: <uzasadnienie>".
2. Notes QUESTIONa zawiera: stary brzmienie, nowe brzmienie, różnica semantyczna, dlaczego wartościowe.
3. Sygnalizuje w czacie ("proponuję zmianę RS nXX — zobacz nYY").
4. Czeka na decyzję usera (akceptuje / odrzuca / modyfikuje).
5. Po akceptacji: edytuje RS, QUESTION → DONE z notatką "zaakceptowane, RS zaktualizowany" + triggeruje ponowny audyt (bo zmiana RS = zmiana ograniczeń = możliwe nowe konflikty).
