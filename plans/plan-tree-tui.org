Wykonalność: trywialna. TreePanel.getSelectedItem() już istnieje i jest aktualizowane przez ruchy klawiatury. main.mts już rejestruje onSelectionChanged dla DetailsPanel — analogiczna ścieżka do MCPServer.
Trzy ścieżki rozważone:
(A) Resource tree://selection — { id, title, path }. CC musi sam zapytać; bez wzmianki w SERVER_INSTRUCTIONS nie pomyśli.
(B) Tool getSelection() — bardziej naturalny dla agenta niż resource, ale nadal pull.
(C) Ambient context — dokleić selection: {id, title, path} do KAŻDEJ odpowiedzi runTool i każdego resource. CC i tak woła tools przy akcjach, więc dostaje selekcję za darmo bez pytania.
Rekomendacja: C jako podstawa (ambient) + B jako opcjonalny tool dla celowych pytań bez akcji ("co jest zaznaczone?"). A pomijać — duplikuje B.
Edge case: węzeł skasowany przed odczytem — bezpiecznie zwracać null.

(A) Property :PRIORITY: high|medium|low — czyste semantycznie, swobodna skala. Niewidoczne w pliku bez otwarcia drawera. Wymaga setProperty/getProperty (już istnieją).

** DONE Decyzja: paleta = akcent + neutral (czerwony high, pomarańczowy med, bez koloru low/none) — opcja C
:PROPERTIES:
:ID: n27
:END:
Opcje:
(A) Klasyczny heat: czerwony (high) → żółty (med) → zielony (low). Intuicyjny, ale "zielony" mylące — sugeruje "ok/done".
(B) Saturacja: nasycony czerwony (high) → bladoróżowy (med) → szary (low). Spójny ton, niższy priorytet = wycofany.

Rekomendacja: (C) lub (D). Kolor tylko tam gdzie naprawdę woła o uwagę.
